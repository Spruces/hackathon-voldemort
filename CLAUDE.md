> 변경이 필요하면 사용자(기획자)와 합의 후 수정한다.

## 1. 프로젝트 컨텍스트

- **상황**: 해커톤 진행 중. 팀원이 **Git 저장소를 통해** 같은 폴더의 문서·코드를 공유하며 작업한다. 즉 내가 수정·커밋·푸시한 내용은 곧바로 다른 팀원에게 전달된다.
- **주제**: 사장님 식당 DB(228건)를 VIP 임원에게 그룹별 URL+4자리 비번으로 제공하는 모바일 우선 대시보드.
- **핵심 산출물(현재)**:
  - [요구사항 정의서.md](요구사항%20정의서.md) — 메인 명세 (현재 v1.3)
  - [작업 정리.md](작업%20정리.md) — 경과·분석·다음 작업 후보
  - `Restaurant Korea_May 2026.xlsx` — 원본 데이터 (228건, 시트 `Korea`)
- **방향 전환 결정사항(v1.3, 2026-06-24)**: Bedrock(LLM) → **네이버/카카오 지도 공식 API**, 사장님 편집(FR-10/11) 신설, 신규 맛집 검색 조회 전용(FR-12), 크롤링 미채택, 예약·주차 사장님 수동 입력. 상세는 요구사항 정의서 **00장**.

## 2. 사용자(나) 역할

- **나는 기획자**다. 코드는 직접 작성하지 않는다.
- 나의 작업 범위: **요구사항 정의·문서화·UX 라이팅·발표 시나리오·정책 결정**.
- 개발자가 별도로 있고, 코드는 그쪽이 담당한다. **나는 코드 리뷰어가 아니다**.
- 따라서 응답할 때:
  - **개발자 전문용어 최소화**, 필요한 경우 한 줄 설명을 곁들인다.
  - 코드 변경이 필요한 작업이라면 *"개발자에게 전달할 항목"* 형태로 정리해 준다.
  - 데이터 분석·문서 정리·표 생성 같은 작업은 적극적으로 도와준다.

## 3. 문서·코드가 꼬이지 않게 하는 규칙

> 해커톤은 빠르게 변하므로, 문서 일관성이 깨지면 팀 전체가 헛수고를 한다. 아래 규칙은 그 방지선이다.

### 3-1. 변경 전 항상 확인할 것

1. 문서를 수정하기 전에 **현재 상태를 먼저 읽고** 어디를 어떻게 바꿀지 한 줄로 알려준 뒤 진행한다.
2. **요구사항 정의서·작업 정리.md**는 팀 공유 문서다. 임의로 큰 폭 수정하지 말고, 변경 의도를 사용자에게 먼저 확인한다.
3. 새 파일을 만들기 전에 **기존 문서에 추가하는 게 더 적합한지** 먼저 따진다. (사용자가 명시적으로 새 파일을 요청한 경우에만 신규 생성)

### 3-2. 버전·변경 이력 관리

- **요구사항 정의서.md**는 의미 있는 변경 시 **상단 헤더의 버전(`v1.x`)과 09장 변경 이력**을 함께 갱신한다.
- 변경 요약은 **00장(개발자 공유용)**에 표 형식으로 누적한다 — 기존 표를 지우지 말고 행을 추가하는 방식.
- **작업 정리.md**는 작업 경과 표(1번 항목)에 단계가 추가될 때만 갱신한다. 해커톤 중에는 매일 한 줄 정도가 적당.

### 3-3. 데이터·정책 결정의 단일 진실 소스(SSOT)

- **업종 정규화(7종 매핑)·위치 정규화·메모 분해 규칙**은 [요구사항 정의서.md 07장](요구사항%20정의서.md#07-데이터-현황-분석-및-보완-실제-소스-기반)이 SSOT다.
- 다른 문서에 같은 표를 복제하지 말고, **링크로 참조**한다. 복제하면 어느 한 쪽만 바뀌어 충돌이 난다.
- 정책이 바뀌면 SSOT(요구사항 정의서)부터 고치고, 다른 문서는 링크만 둔다.

### 3-4. 코드 영역 손대지 않기 (현재 단계)

- 현재 레포에는 **아직 애플리케이션 코드가 없다** (문서·엑셀만 있음).
- 코드가 추가된 이후에도, 사용자(기획자)가 명시적으로 코드 작업을 요청하지 않는 한 **소스 코드는 수정하지 않는다**.
- 개발자가 코드를 짜고 있을 가능성이 있으므로, 임의로 파일을 만들거나 디렉터리 구조를 잡지 않는다.

### 3-5. Git 협업 규칙 (★ 핵심 — 팀원과 충돌 방지)

> 이 레포는 **팀원들과 Git으로 실시간 공유**된다. 잘못 푸시하면 모든 팀원이 영향을 받는다.

- **커밋·푸시는 사용자가 명시적으로 요청했을 때만** 실행한다. 자동 커밋·자동 푸시 금지.
- 작업 시작 전, 의심스러우면 `git status` / `git log --oneline -5`로 **현재 상태를 먼저 확인**한다 (팀원이 방금 푸시했을 수 있음).
- 큰 변경이라면 작업 전에 `git pull`을 권장한다 — 사용자에게 *"먼저 pull 받으시겠어요?"* 라고 물어본다.
- 커밋 메시지는 한국어 `정의서 v1.x 개정: ...` / `초기 커밋: ...` 형식의 기존 컨벤션을 따른다 (현재 main 브랜치 커밋 참고).
- **절대 자율적으로 하지 않을 것**: 강제 푸시(`push --force`), 브랜치 삭제, `reset --hard`, `checkout --` (uncommitted 변경 폐기), `clean -f`, 다른 팀원 커밋 amend.
- **민감 파일 커밋 금지**: API 키·비밀번호·`.env` 같은 파일은 절대 add 하지 않는다. 지도 API 키가 들어오면 `.gitignore` 먼저 확인.
- **엑셀 원본(`Restaurant Korea_May 2026.xlsx`) 변경**은 사장님 데이터의 진실 소스이므로 **수정·재생성하지 않는다** (팀 전체에 영향). 분석은 read-only.
- 이미 푸시된 커밋은 amend 하지 않는다 — 새 커밋을 추가하는 방식으로 수정.

## 5. 응답 스타일

- **언어**: 한국어로 답한다. 사용자가 영어로 물으면 영어로.
- **길이**: 짧고 명확하게. 기획 결정에 필요한 정보만.
- **형식**: 표·리스트를 적극 활용. 요구사항 정의서가 표 기반이라 일관성을 유지.
- **추측 금지**: 데이터 수치·결정 사항은 **소스(엑셀·문서)를 직접 확인하고 인용**한다. 기억에 의존해 추정하지 않는다.
- **모를 때**: "확인 필요"라고 명시하고 사용자에게 결정을 요청한다. 임의 결정으로 문서를 채우지 않는다.

## 7. 금지·주의 (Do Not)

- ❌ 사용자 확인 없이 요구사항 정의서의 **장(章) 구조나 ID 체계(FR-xx, OOS-xx)** 변경
- ❌ 같은 정보를 여러 문서에 복제 (SSOT 위반)
- ❌ 변경 이력(09장) 갱신 없이 본문만 수정
- ❌ 기획자에게 코드 스니펫만 던지고 끝내기 — 항상 **무엇을 왜** 바꾸는지 한국어 설명 동반
- ❌ 임의로 새 파일·폴더 생성 (특히 README, 설계 문서, 계획 문서 등은 요청 시에만)
- ❌ 영어로 된 식당 데이터를 임의 번역하거나, 결측치를 LLM 추정값으로 채우기 (v1.3 결정에 위배)
- ❌ **자율적인 git commit / push / force push / reset --hard / 브랜치 삭제** — 팀원 작업 손실 위험
- ❌ **API 키·비밀번호 같은 민감정보 커밋** — 공개 저장소면 즉시 노출됨
