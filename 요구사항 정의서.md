## 01. 프로젝트 개요

### 목적
사장님이 보유한 식당 데이터베이스를 웹 기반 대시보드로 시각화하여, 그룹별 전용 URL로 초대된 C-Level 임원들이 언제 어디서나 식당 정보를 탐색하고 활용할 수 있도록 한다.

### 사용자 (Actor)

| 역할 | 설명 |
|------|------|
| **Owner (사장님)** | 시스템 관리자. 식당 DB 소유자. 그룹 생성·관리 및 식당 배정 주체. 모든 그룹 접근 가능. |
| **C-Level (임원)** | 그룹 전용 URL + 4자리 비밀번호로 접근. 해당 그룹의 식당 정보 조회·탐색만 가능. |

### 핵심 전제

- 외부 식당 정보는 **AWS Bedrock**을 통해 보강
- 식당 목록의 필터/분류는 사장님 DB 기반 룰베이스 동작
- 웹뷰(Web View) 형태로 제공. 앱 스토어 배포 없음
- 그룹별 전용 URL + 4자리 비밀번호로 접근 제어
- VIP 대상 서비스로 고급스러운 디자인 방향성 유지

> 🔎 **데이터 기준**: 현재 소스 데이터는 단일 시트(`Korea`) **228개 식당**. 전 식당이 한국(서울 강남권 중심) 소재.
> 시트명이 국가 단위(`Korea`)이므로 향후 다국가 확장 가능성을 데이터 모델에 반영 권장(05장 참고).

## 02. 사용자 요구사항

### 2-1. 식당 탐색 · 조회

| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| FR-01 | 업종별 탭 필터 | 한식, 일식, 양식, 중식, 이탈리안 등 업종 카테고리를 탭/칩 형태로 제공. 선택 시 해당 업종 식당만 리스트에 표시. DB에 저장된 업종 구분 데이터 기반으로 동작 (룰베이스). | 필수 |
| FR-02 | 식당 리스트 뷰 | 필터 결과를 카드 형태로 표시. 각 카드에는 식당명, 업종, 위치(동), 별점, 예약 가능 여부, 대표 이미지 포함. | 필수 |
| FR-03 | 식당 상세 조회 | 리스트 카드 클릭 시 상세 화면 진입. 상세 정보: 식당명(한/영), 위치, 업종, 전화번호, 주소, 메모(사장님 DB), 별점, 리뷰, 예약 가능 여부, 메뉴 (AWS Bedrock 보강 항목). | 필수 |
| FR-04 | 키워드 검색 | 식당명, 위치(동), 업종 기준 텍스트 검색. 실시간 필터링 (입력 즉시 결과 갱신). DB 내부 검색이므로 외부 API 호출 없음. | 필수 |
| FR-05 | 복합 필터 | 업종 외 추가 필터 제공: 지역(동), 예약 가능 여부, 별점 순 정렬. 필터 조건 중복 선택 가능. 초기화 버튼 제공. | 권장 |

> 🔎 **FR-01 데이터 기준 — 업종 정규화 필수**:
> 소스의 `종류` 컬럼은 **34종**으로, 영문/국문 혼용 + 세부 분류가 난립해 있어(`Korean`·`한식`·`고기집`·`돼지고기`·`닭고기` 등) **그대로는 탭 필터로 사용 불가**.
> → 정규화 매핑 테이블을 두어 **대표 업종 7종**으로 묶는 룰을 정의해야 함 (07-2 매핑표 참고).
>
> 🔎 **FR-02 / FR-03 데이터 기준 — 없는 항목**:
> `별점`, `리뷰`, `예약 가능 여부`, `메뉴`, `대표 이미지`는 **소스 DB에 컬럼 자체가 없음**.
> → 카드/상세에서 이 항목들은 (1) Bedrock 보강 전까지 **빈 상태/스켈레톤 처리**, (2) 대표 이미지는 **플레이스홀더** 필요(04장).
>
> 🔎 **FR-04 데이터 기준**: 검색 대상 텍스트가 한글·영문·한자(주소) 혼재. 식당명(영문) 95% 채워짐(11곳 누락) → 영문 검색 시 **한글명 fallback** 필요.

### 2-2. 그룹 접근 제어 · 인증

| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| FR-06 | 그룹별 전용 URL 발급 | Owner가 그룹을 생성하면 해당 그룹 전용 URL(또는 QR코드)이 발급된다. URL 자체가 그룹 식별자 역할을 하며, 해당 URL로만 그룹의 식당 데이터에 접근 가능. QR코드는 다운로드·공유 가능한 형태로 제공. | 필수 |
| FR-07 | 4자리 숫자 비밀번호 인증 | 각 그룹 URL 접속 시 4자리 숫자 비밀번호 입력 화면이 표시된다. 비밀번호는 그룹별로 개별 설정되며 Owner가 관리. 인증 성공 시 해당 그룹의 식당 리스트 진입. 세션 유지로 재방문 시 재입력 최소화. | 필수 |
| FR-08 | 그룹 관리 (Owner 전용) | Owner는 별도 관리 화면에서 그룹 생성·삭제, 그룹별 비밀번호 변경, URL/QR코드 재발급, 그룹에 포함할 식당 수동 배정이 가능. Owner 계정은 모든 그룹에 접근 가능. | 필수 |

> 🔎 **데이터 기준 — 그룹/인증 데이터는 소스에 전무**: 엑셀에는 그룹·비밀번호·식당↔그룹 매핑 정보가 **없음**.
> → 그룹(Group), 그룹-식당 매핑(GroupRestaurant), 비밀번호(해시), 세션 토큰은 **신규 테이블로 설계**해야 함(07-4 데이터 모델 제안).
> → URL 식별자는 추측 불가능한 토큰(예: 22자 nanoid/UUID) 사용. 비밀번호 4자리는 **해시 저장 + 무차별 대입 방지**(FR-07 보완 항목, 05장 보안).

### 2-3. AWS Bedrock 연동 (식당 정보 보강)

| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| FR-09 | 외부 데이터 보강 | 사장님 DB에 없거나 비어있는 항목(별점, 리뷰, 메뉴, 예약 가능 여부)을 AWS Bedrock을 통해 조회·보완. 식당 상세 화면 진입 시 비동기 로딩. | 권장 |

> 🔎 **데이터 기준 — 보강 대상이 사실상 전부**: 소스에 별점·리뷰·메뉴·예약여부가 0건이므로,
> 이 항목들은 "비어있는 항목 보완"이 아니라 **상세 진입 시 매번 Bedrock 호출이 기본 동작**이 됨.
> → 비용·지연 관리를 위해 **보강 결과 캐싱(TTL)** 및 **호출 실패 시 graceful fallback**(빈 상태 표기) 요구사항 추가(07-3).
> → 주소/전화번호는 소스에 99~100% 존재하므로 Bedrock은 **정규화·최신화 용도(선택)** 로만 사용.

## 03. UX 방향 정의

### 그룹 접근 플로우 — URL + 4자리 비밀번호

```
그룹 URL 접속 / QR 스캔  →  4자리 비밀번호 입력  →  해당 그룹 식당 리스트  →  탐색 · 상세 조회
```

- 그룹 간 데이터는 독립. A그룹 URL로 B그룹 데이터 접근 불가
- Owner는 별도 관리 URL로 모든 그룹 접근 가능
- 세션 유지로 재방문 시 비밀번호 재입력 최소화

---

### 업종 탭 필터 — 핵심 인터랙션

```
전체 보기 (기본)  →  업종 탭 선택  →  해당 업종 리스트만 표시  →  카드 클릭 → 상세
```

- DB 내 업종(종류) 컬럼 기반 룰베이스 동작
- 실시간 API 호출 없이 클라이언트 필터링 또는 간단한 서버 쿼리로 처리 가능
- 탭은 가로 스크롤 가능, 업종 수에 따라 유동적으로 확장

> 🔎 **데이터 기준 — 권장 탭 구성(정규화 후, 228건)**:
> `한식(68)` · `일식(43)` · `이탈리안(33)` · `중식(30)` · `프렌치(29)` · `아메리칸·양식(20)` · `아시안(5)`
> → 7개 탭 + `전체` 기본. 가로 스크롤로 충분히 수용 가능. 매핑 규칙은 07-2 참고.

> 🔎 **메모 컬럼 재정의 필요**: 소스의 메모(notes)는 "Owner 비공개"로 정의되어 있으나 실제 내용은
> **① 공개 가치가 있는 설명**(예: *"Best Italian in 서초동"*, *"미슐랭가이드 2019"*),
> **② 영업시간/휴무**(예: *"17:30-24:00, 월화 휴무"* — 약 27건),
> **③ 예약 방식**(예: *"캐치테이블 예약 only, 전화 불가"* — 약 5건),
> **④ 추천인/내부 메모**(예: *"Dorothee (BNP) 추천"*) 가 혼재.
> → 메모를 **`public_desc`(공개 설명) / `hours`(영업시간) / `internal_memo`(Owner 비공개)** 로 분해하거나,
> 최소한 **공개/비공개 플래그**를 두어 임원 화면 노출 범위를 명확히 할 것.

## 05. 비기능 요구사항 · 제약사항

| 항목 | 내용 |
|------|------|
| **플랫폼** | 웹뷰(Web View) 전용. 앱 스토어 배포 없음. PC·태블릿·모바일 브라우저 지원 (반응형). |
| **보안** | 그룹별 URL + 4자리 비밀번호 인증. 그룹 간 데이터 격리. HTTPS 필수. **비밀번호 해시 저장, 그룹별 로그인 시도 제한(rate limit)으로 4자리(10,000조합) 무차별 대입 방지.** |
| **인프라** | AWS 기반 (Bedrock 연동). 기존 PostgreSQL DB 연결. S3 이미지 스토리지 활용 가능. |
| **성능** | 리스트 초기 로딩 3초 이내. Bedrock 보강 데이터는 상세 진입 시 비동기 로딩 허용. **데이터 규모가 228건 수준이므로 전체 클라이언트 사이드 필터/검색이 충분히 가능(서버 부하 낮음).** |
| **디자인** | VIP 대상 프리미엄 UI. 다크 테마 기본. |
| **접근성** | 그룹 URL + 비밀번호 보유자만 접근 가능한 클로즈드 서비스. 공개 가입 없음. |
| **데이터 품질** | 🔎 업종·위치 표기 정규화, 영문명/전화번호 결측 보완을 **데이터 적재(ETL) 단계에서 1회 처리** 후 운영(07장). |
| **국제화(향후)** | 🔎 소스 시트명이 `Korea` → 다국가 확장 대비, 식당 테이블에 `country`/`city` 컬럼 여지 확보. |

## 08. 프로덕션 전환 요건 (Production Readiness)

> 해커톤/데모 범위와 별개로, **실제 제품(서비스)으로 운영하려면** 충족해야 하는 요건.
> 우선순위 P0(출시 차단 요소) → P2(단계적 개선) 순. P0는 미충족 시 제품 출시 불가 수준.

### 08-1. AI 보강 데이터 신뢰성 (P0) — 본 제품 최대 리스크

소스에 별점·리뷰·메뉴·예약여부가 **0/228**이므로, FR-09 보강 데이터는 사실상 **LLM 생성값을 VIP에게 사실처럼 제시**하는 구조다. LLM은 별점·전화번호·메뉴를 그럴듯하게 **환각(hallucination)** 할 수 있어, 잘못된 정보는 곧 제품 신뢰 붕괴로 이어진다.

| 요건 | 내용 |
|------|------|
| 출처/신뢰도 라벨 | Owner DB 값과 AI 추정값을 **시각적으로 구분**. "AI 추정 · 확인 필요" 등 provenance 표기 필수. |
| 검증 가능한 소스 우선 | 별점·리뷰는 LLM 단독 생성보다 **실제 출처(지도/예약 플랫폼 등)를 Bedrock으로 요약**하는 구조 권장. 순수 생성값이면 면책 문구 + "최종 확인은 전화로" 안내. |
| 보수적 기본값 | 불확실하면 **빈칸 유지**가 틀린 값 노출보다 낫다(원칙: "자신있게 틀리지 않기"). |
| 정확도 모니터링 | 보강 결과 샘플 검수 루프, 오류 신고 경로. |

### 08-2. 접근 보안 (P0) — 4자리 비밀번호는 데모용

공유 URL + 4자리(1만 조합)는 기밀 VIP 데이터 보호 수단으로 약하다.

| 요건 | 내용 |
|------|------|
| 무차별 대입 방지 | 그룹별 **시도 횟수 제한·잠금·지수 백오프**. (미적용 시 1만 조합은 자동화로 분 단위 돌파) |
| 비밀번호 해시 저장 | bcrypt/argon2. 평문/단순 해시 금지. |
| URL 토큰 엔트로피 | 추측 불가 토큰(예: nanoid 22자/UUID). 순차·예측 가능 ID 금지. |
| 세션 보안 | 쿠키 `HttpOnly`·`Secure`·`SameSite`, 만료·갱신 정책. |
| 접근 감사 로그 | 누가·언제·어느 그룹에 접근했는지 기록(기밀 서비스 필수). |
| 인증 방식 재검토(P1) | 4자리 대신 **매직링크 / 6자리 + 짧은 만료**가 제품 등급에 적합한지 검토. |

### 08-3. 데이터 운영 파이프라인 (P1) — "엑셀 1회 적재"에서 탈피

식당 데이터는 계속 변한다(폐업·번호·영업시간·이전 — 소스 메모에 "주소 변경됨" 흔적 존재).

| 요건 | 내용 |
|------|------|
| 데이터 관리 수단 | Owner CRUD UI **또는** 검증 포함 엑셀 재업로드 파이프라인. 단일 진실 공급원(SSOT) 확정. |
| 정규화 매핑 운영화 | 업종·위치 매핑(07-2/07-3)을 **하드코딩이 아닌 운영 가능한 테이블**로. 미매핑 업종 유입 시 처리 규칙 정의. |
| 임포트 유효성 검사 | 적재 시 결측·중복·형식 검증(영문명 11건·전화 2건 결측 사례). |
| 데이터 버전/이력 | 변경 이력 추적, 롤백 가능성. |
