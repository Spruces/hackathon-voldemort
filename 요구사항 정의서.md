## 01. 프로젝트 개요

### 목적
사장님이 보유한 식당 데이터베이스를 모바일 웹 대시보드로 시각화하여, 전용 URL로 초대된 C-Level 임원들이 언제 어디서나 식당 정보를 탐색하고 활용할 수 있도록 한다.

### 사용자 (Actor)

| 역할 | 설명 |
|------|------|
| **Owner (사장님)** | 시스템 관리자. 식당 DB 소유자. 메모·카테고리 편집, 비밀번호·URL 관리 주체. |
| **C-Level (임원)** | 전용 URL + 6자리 비밀번호로 접근. **전체 식당(228건)** 정보 조회·탐색만 가능(편집 불가). |

### 핵심 전제

- 외부 식당 정보는 **네이버/카카오 지도 공식 API**를 통해 보강 (v1.3: Bedrock에서 전환)
- 공식 API로 얻지 못하는 항목(예약 가능 여부·주차 등)은 **사장님이 메모/필드로 수동 입력**
- **크롤링은 채택하지 않음** (약관·법적 리스크) — 08-5 참고
- 식당 목록의 필터/분류는 사장님 DB 기반 룰베이스 동작
- 사장님은 식당별 **메모 추가·카테고리(분류) 편집** 가능. 임원(C-Level)은 조회 전용
- 사장님은 지도에서 **DB에 없는 신규 맛집 검색** 가능 (조회 전용, 저장 안 함)
- **모바일 우선** 웹뷰(Web View) 형태로 제공. 앱 스토어 배포 없음 (모바일 브라우저로 접근)
- 전용 URL + **6자리 비밀번호**로 접근 제어. 모든 임원은 동일한 전체 목록 열람(그룹 차등 없음, v1.3)
- VIP 대상 서비스로 고급스러운 디자인 방향성 유지

> 🔎 **데이터 기준**: 현재 소스 데이터는 단일 시트(`Korea`) **228개 식당**. 전 식당이 한국(서울 강남권 중심) 소재.
> 시트명이 국가 단위(`Korea`)이므로 향후 다국가 확장 가능성을 데이터 모델에 반영 권장(05장 참고).

### 2-3. 지도 API 연동 (식당 정보 보강) — v1.3 개정

> v1.2의 AWS Bedrock(LLM) 보강을 **네이버/카카오 지도 공식 API 보강 + 사장님 수동 입력**으로 전환.
> 사용 지도 API(네이버/카카오)는 무료 할당량·연동 난이도를 보고 **개발자가 선택**.

| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| FR-09 | 지도 API 데이터 보강 | 식당 상세 진입 시 지도 공식 API로 **위치 좌표·정규화 주소·전화번호·카테고리·영업시간** 등 공식 제공 항목을 조회·보완. 비동기 로딩. | 권장 |
| FR-09a | 지도/길찾기 표시 | 식당 위치를 지도에 핀으로 표시하고, 외부 지도 앱 길찾기로 연결. | 권장 |
| FR-12 | 신규 맛집 검색(조회 전용) | 사장님이 DB에 없는 식당을 지도 API로 검색해 위치·기본정보를 **확인만** 가능. **DB 저장은 하지 않음**(Phase 2 검토). | 권장 |

> 🔎 **데이터 기준 — 공식 API가 주는 것 vs 못 주는 것**:
> 지도 공식 API는 통상 **위치·주소·전화·카테고리·영업시간**을 제공하나,
> **별점·리뷰·예약 가능 여부·주차 여부**는 제공이 없거나 제한적임(약관상 크롤링도 불가).
> → 별점/리뷰는 **표시하지 않거나** API가 공식 제공하는 범위 내에서만 사용.
> → **예약 가능 여부·주차 여부는 사장님이 수동 입력**(FR-10 메모/필드)으로 채운다 — 본 제품의 정보 신뢰성 핵심.
> → 보강 결과는 **캐싱(TTL)** 하고, API 실패 시 **graceful fallback**(빈 상태 표기)(07-3).
> → 주소/전화번호는 소스에 99~100% 존재하므로 지도 API는 **정규화·최신화·좌표 획득 용도**로 활용.

### 2-4. 사장님 직접 편집 (Owner 전용) — v1.3 신설

> 사장님만 편집 가능. 임원(C-Level)은 조회 전용(OOS-03 유지). 모든 편집은 Owner 인증 후에만 허용.

| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| FR-10 | 메모·부가정보 편집 | 식당별로 정보를 추가/수정/삭제. ① **메모(자유 텍스트)는 Owner 전용 — 임원에게 노출하지 않음**(2026.06.24 확정). ② 식당 소개·영업시간·**예약 가능 여부·주차 여부**(지도 API 미제공분)는 사장님이 입력하며 **임원에게 표시**. | 필수 |
| FR-10a | 노출 범위 표기(배지) | 편집·상세 화면의 각 항목 옆에 **🔒(나만 보기) / 📢(공유)** 배지(아이콘+텍스트)를 표시해, 사장님이 어떤 정보가 임원에게 보이는지 한눈에 알 수 있게 함. 임원 화면에는 🔒 항목을 아예 렌더링하지 않음. | 필수 |
| FR-11 | 카테고리(분류) 편집 | 식당의 대표 업종(07-2의 7종) 및 사용자 정의 분류/태그를 사장님이 지정·변경. 신규 업종 유입 시 매핑 추가. | 필수 |

> 🔎 **데이터 기준 — 04장 메모 분해와 연계**: 소스의 메모(notes)는 ① 식당 소개(`public_desc`), ② 영업시간(`hours`), ③ 내부 메모(`internal_memo`)가 혼재.

### 07-4. 신규 데이터 모델 제안 (그룹·인증·보강)

소스에 없는 영역으로, 아래 테이블 신규 설계 필요:

```
Restaurant      (id, name_kor, name_eng, location_norm, location_raw,
                 category_norm, category_raw, tel, address, lat, lng,
                 public_desc, hours, internal_memo, image_url, country,
                 reservable, parking, updated_by, updated_at)
AccessConfig    (id, url_token, password_hash, session_days, updated_at)
                 ← 단일 접근 설정(URL 토큰 + 6자리 비번 해시 + 세션 기간)
MapCache        (restaurant_id, map_place_id, address_norm, tel,
                 hours, category, lat, lng, fetched_at, ttl)  ← 지도 API 보강 캐시
Session         (token, expires_at)                        ← FR-07 세션 유지(기본 7일)
```

- **v1.3 단순화**: 그룹 차등 노출 미채택 → `Group`·`GroupRestaurant` 테이블 **삭제**. 접근 제어는 `AccessConfig` 단일 행으로 충분.
- **예약여부(`reservable`)·주차(`parking`)** 는 사장님 수동 입력 필드(FR-10). `updated_by/updated_at`으로 편집 이력 최소 추적.
- **지도 API 보강**(주소·전화·영업시간·좌표)은 `MapCache`에 캐싱하고 `fetched_at + ttl`로 만료 관리 → FR-09 호출·지연 완화.
- **별점/리뷰**는 지도 API가 공식 제공하는 범위에서만 표시(LLM 생성 미사용). 미제공 시 미표시.
- **메모 분해**: `public_desc`(임원 노출) / `hours` / `internal_memo`(🔒 Owner 전용)로 분리 → 04장 보완 사항 반영.
- **신규 맛집 검색(FR-12)** 은 조회 전용이므로 별도 저장 테이블 불필요(지도 API 응답을 화면에 직접 표시).

### 07-5. ETL(데이터 적재) 처리 항목 — 운영 전 1회

1. 헤더/공란 행 제거, 228건 정제 적재
2. `종류` → `category_norm` 매핑(07-2)
3. `위치` → `location_norm` 정규화(07-3)
4. 메모 → `public_desc` / `hours` / `internal_memo` 분해(04장)
5. 영문명 11건·전화번호 2건 결측 → 지도 API 또는 수기 보완
6. 대표 이미지 `image_url` 컬럼 추가(초기값: 플레이스홀더)

## 08. 프로덕션 전환 요건 (Production Readiness)

> 해커톤/데모 범위와 별개로, **실제 제품(서비스)으로 운영하려면** 충족해야 하는 요건.
> 우선순위 P0(출시 차단 요소) → P2(단계적 개선) 순. P0는 미충족 시 제품 출시 불가 수준.

### 08-1. 보강 데이터 신뢰성 (P0) — v1.3에서 리스크 대폭 완화

> ✅ **v1.3 변경 효과**: LLM(Bedrock) 생성값을 사실처럼 제시하던 구조를 폐기하고, **지도 공식 API(검증 가능 출처) + 사장님 수동 입력**으로 전환.
> 환각(hallucination) 리스크는 **상당 부분 해소**됨. 다만 출처 구분·결측 처리 원칙은 계속 유효.

| 요건 | 내용 |
|------|------|
| 출처 라벨 | Owner DB 값 / 지도 API 보강값 / 사장님 입력값을 **시각적으로 구분**. provenance(출처) 표기 권장. |
| 검증 가능한 소스 우선 | 별점·리뷰는 **지도 API가 공식 제공하는 범위에서만** 사용. LLM 단독 생성값은 **사용하지 않음**. |
| 보수적 기본값 | 불확실하면 **빈칸 유지**가 틀린 값 노출보다 낫다(원칙: "자신있게 틀리지 않기"). 예약·주차 미입력 시 "정보 없음" 표기. |
| 입력값 신뢰 | 예약·주차 등 사장님 수동 입력값은 최신성 관리(편집 일시 표기 권장). |

### 08-2. 접근 보안 (P0) — 최우선 항목

> v1.3에서 **6자리(100만 조합)** 로 강화했으나, 공유 URL + 비밀번호 단일 방식은 여전히 기밀 데이터에 대해 강화 여지가 있음. 아래는 출시 전 충족 요건.

| 요건 | 내용 |
|------|------|
| 무차별 대입 방지 | **시도 횟수 제한·잠금·지수 백오프** 필수. (미적용 시 100만 조합도 자동화로 돌파 가능) |
| 비밀번호 해시 저장 | bcrypt/argon2. 평문/단순 해시 금지. |
| URL 토큰 엔트로피 | 추측 불가 토큰(예: nanoid 22자/UUID). 순차·예측 가능 ID 금지. |
| 세션 보안 | 쿠키 `HttpOnly`·`Secure`·`SameSite`, 만료(기본 7일)·갱신·강제 만료 정책. |
| 접근 감사 로그 | 누가·언제 접근했는지 기록(기밀 서비스 필수). |
| 인증 방식 재검토(P1) | 6자리 비번 대신 **매직링크 / OTP**가 제품 등급에 적합한지 검토. |

### 08-3. 데이터 운영 파이프라인 (P1) — "엑셀 1회 적재"에서 탈피

식당 데이터는 계속 변한다(폐업·번호·영업시간·이전 — 소스 메모에 "주소 변경됨" 흔적 존재).

| 요건 | 내용 |
|------|------|
| 데이터 관리 수단 | Owner CRUD UI **또는** 검증 포함 엑셀 재업로드 파이프라인. 단일 진실 공급원(SSOT) 확정. |
| 정규화 매핑 운영화 | 업종·위치 매핑(07-2/07-3)을 **하드코딩이 아닌 운영 가능한 테이블**로. 미매핑 업종 유입 시 처리 규칙 정의. |

| 데이터 버전/이력 | 변경 이력 추적, 롤백 가능성. |
