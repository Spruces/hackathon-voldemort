## 01. 프로젝트 개요

### 목적
사장님이 보유한 식당 데이터베이스를 모바일 웹 대시보드로 시각화하여, 전용 URL로 초대된 C-Level 임원들이 언제 어디서나 식당 정보를 탐색하고 활용할 수 있도록 한다.

### 사용자 (Actor)

| 역할 | 설명 |
|------|------|
| **Owner (사장님)** | 시스템 관리자. 식당 DB 소유자. 메모·카테고리 편집, 비밀번호·URL 관리 주체. |
| **C-Level (임원)** | 전용 URL + 6자리 비밀번호로 접근. **전체 식당(228건)** 정보 조회·탐색만 가능(편집 불가). |

### 핵심 전제

- 외부 식당 정보는 **네이버/카카오 지도 공식 API**를 통해 보강 (v1.3: Bedrock에서 전환)
- 공식 API로 얻지 못하는 항목(주차 등)은 **사장님이 메모/필드로 수동 입력**
- **크롤링은 채택하지 않음** (약관·법적 리스크) — 08-5 참고
- 식당 목록의 필터/분류는 사장님 DB 기반 룰베이스 동작
- 사장님은 식당별 **메모 추가·카테고리(분류) 편집** 가능. 임원(C-Level)은 조회 전용
- 사장님은 지도에서 **DB에 없는 신규 맛집 검색** 가능 (조회 전용, 저장 안 함)
- **모바일 우선** 웹뷰(Web View) 형태로 제공. 앱 스토어 배포 없음 (모바일 브라우저로 접근)
- 전용 URL + **6자리 비밀번호**로 접근 제어. 모든 임원은 동일한 전체 목록 열람(그룹 차등 없음, v1.3)
- VIP 대상 서비스로 고급스러운 디자인 방향성 유지

> 🔎 **데이터 기준**: 현재 소스 데이터는 단일 시트(`Korea`) **228개 식당**. 전 식당이 한국(서울 강남권 중심) 소재.
> 시트명이 국가 단위(`Korea`)이므로 향후 다국가 확장 가능성을 데이터 모델에 반영 권장(05장 참고).

### 2-3. 지도 API 연동 (식당 정보 보강) — v1.3 개정

> v1.2의 AWS Bedrock(LLM) 보강을 **네이버/카카오 지도 공식 API 보강 + 사장님 수동 입력**으로 전환.
> 사용 지도 API(네이버/카카오)는 무료 할당량·연동 난이도를 보고 **개발자가 선택**.

| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| FR-09 | 지도 API 데이터 보강 | 식당 상세 진입 시 지도 공식 API로 **위치 좌표·정규화 주소·전화번호·카테고리·영업시간** 등 공식 제공 항목을 조회·보완. 비동기 로딩. | 권장 |
| FR-09a | 지도/길찾기 표시 | 식당 위치를 지도에 핀으로 표시하고, 외부 지도 앱 길찾기로 연결. | 권장 |
| FR-12 | 신규 맛집 검색(조회 전용) | 사장님이 DB에 없는 식당을 지도 API로 검색해 위치·기본정보를 **확인만** 가능. **DB 저장은 하지 않음**(Phase 2 검토). | 권장 |

> 🔎 **데이터 기준 — 공식 API가 주는 것 vs 못 주는 것**:
> 지도 공식 API는 통상 **위치·주소·전화·카테고리·영업시간**을 제공하나,
> **별점·리뷰·예약 가능 여부·주차 여부**는 제공이 없거나 제한적임(약관상 크롤링도 불가).
> → 별점·리뷰·**예약 여부는 화면에서 제외**(2026.06.24 확정, 미채택).
> → **주차 여부는 사장님이 수동 입력**(FR-10 메모/필드)으로 채운다.
> → 보강 결과는 **캐싱(TTL)** 하고, API 실패 시 **graceful fallback**(빈 상태 표기)(07-3).
> → 주소/전화번호는 소스에 99~100% 존재하므로 지도 API는 **정규화·최신화·좌표 획득 용도**로 활용.

### 2-4. 사장님 직접 편집 (Owner 전용) — v1.3 신설

> 사장님만 편집 가능. 임원(C-Level)은 조회 전용(OOS-03 유지). 모든 편집은 Owner 인증 후에만 허용.

| ID | 기능명 | 설명 | 우선순위 |
|----|--------|------|----------|
| FR-10 | 메모·부가정보 편집 | 식당별로 정보를 추가/수정/삭제. ① **메모(자유 텍스트)는 Owner 전용 — 임원에게 노출하지 않음**(2026.06.24 확정). ② 식당 소개·영업시간·**주차 여부**(지도 API 미제공분)는 사장님이 입력하며 **임원에게 표시**. (예약 여부는 미채택) | 필수 |
| FR-10a | 노출 범위 표기(배지) | 편집·상세 화면의 각 항목 옆에 **🔒(나만 보기) / 📢(공유)** 배지(아이콘+텍스트)를 표시해, 사장님이 어떤 정보가 임원에게 보이는지 한눈에 알 수 있게 함. 임원 화면에는 🔒 항목을 아예 렌더링하지 않음. | 필수 |
| FR-11 | 카테고리(분류) 편집 | 식당의 대표 업종(07-2의 7종) 및 사용자 정의 분류/태그를 사장님이 지정·변경. 신규 업종 유입 시 매핑 추가. | 필수 |

> 🔎 **데이터 기준 — 04장 메모 분해와 연계**: 소스의 메모(notes)는 ① 식당 소개(`public_desc`), ② 영업시간(`hours`), ③ 내부 메모(`internal_memo`)가 혼재.
> **확정 노출 규칙(2026.06.24)**: `internal_memo`(자유 메모)는 **Owner 전용**, 임원 화면 노출 금지. `public_desc`·`hours`·`parking`(주차)은 **임원 노출**.
> 즉 "사장님이 자유롭게 쓰는 메모"는 비공개, "식당을 소개하는 정보"는 공개로 분리한다. 데이터 모델은 07-4 참고.

|------|------|
| 출처 라벨 | Owner DB 값 / 지도 API 보강값 / 사장님 입력값을 **시각적으로 구분**. provenance(출처) 표기 권장. |
| 검증 가능한 소스 우선 | 별점·리뷰는 **지도 API가 공식 제공하는 범위에서만** 사용. LLM 단독 생성값은 **사용하지 않음**. |
| 보수적 기본값 | 불확실하면 **빈칸 유지**가 틀린 값 노출보다 낫다(원칙: "자신있게 틀리지 않기"). 주차 미입력 시 "정보 없음" 표기. |
| 입력값 신뢰 | 주차 등 사장님 수동 입력값은 최신성 관리(편집 일시 표기 권장). |

## 10. 사장님 "Buying" 포인트 (설득 논리)
