## 01. 프로젝트 개요

### 목적
사장님이 보유한 식당 DB(228건)를 **시간·예약 가능 여부** 기준으로 탐색하는 모바일 우선 대시보드.
"지금 어디에 있든, 가장 빨리 도착할 수 있고 예약도 가능한 식당"을 즉시 찾아준다.

### 사용자 (Actor)

| 역할 | 설명 |
|------|------|
| **Owner (사장님)** | 시스템 관리자. 식당 DB 소유자. 편집·관리 주체. |
| **C-Level (임원)** | 전용 URL + 6자리 비밀번호로 접근. 조회·탐색만 가능. |

### 핵심 전제

- **지도 + 도달시간이 1순위**. 식당 리스트는 지도의 부속(바텀시트).
- **캐치테이블 API**로 실시간 예약 가능 여부 조회 (228건 중 208건 매칭, 91.2%)
- **카카오 모빌리티 길찾기 API**로 실시간 교통 반영 도달시간(ETA) 계산
- 업종(7종)·지역(동)은 보조 필터로 유지
- 모바일 우선, 전용 URL + 6자리 비번 인증, 2개 모드(임원 조회/사장님 편집)

> 🔎 **데이터 기준**: `Restaurant Korea_May 2026.xlsx`, **228개 식당**. 서울 강남·용산권 집중.
> 캐치테이블 매칭: 208건(91.2%), 미매칭 20건(8.8%).

## 03. UX 방향 정의

### 메인 플로우 — 지도 + 시간 중심

```
전용 URL 접속 → 비번 입력 → 모드 분기
                              ↓
                    ┌─────────────────────────────────┐
                    │  S2. 메인 (지도 + 바텀시트)       │
                    │                                  │
                    │  [지도] 현위치 + 식당 핀          │
                    │  ├ 핀 색상: 예약가능(골드) / 불가(회색) │
                    │  └ 핀 탭 → 간략 정보 팝업        │
                    │                                  │
                    │  [상단 바] 날짜/시간 선택 + "지금 바로" │
                    │  [필터 칩] 업종 | 지역 | 인원수    │
                    │                                  │
                    │  [바텀시트] 시간순 식당 리스트     │
                    │  ├ 도달시간(분) + 식당명 + 업종   │
                    │  ├ 예약 가능 여부 배지            │
                    │  └ 스와이프 올려서 전체 리스트     │
                    └─────────────────────────────────┘
                              ↓ 카드 탭
                    ┌─────────────────────────────────┐
                    │  S3. 식당 상세                    │
                    │  ├ 도달시간·경로 지도             │
                    │  ├ 예약 가능 날짜·인원            │
                    │  ├ 기본정보(영업시간·주차·가격대)  │
                    │  └ 액션: 예약·길찾기·전화·공유    │
                    └─────────────────────────────────┘
```

### "지금 바로" 모드

```
[지금 바로] 버튼 탭
    ↓
현재 시각 기준:
  ① 영업 중인 식당만 (영업시간 or 캐치테이블 상태=OPEN)
  ② 도달시간 짧은 순 정렬
  ③ 지도에서 가까운 식당 핀 강조
```

### 날짜/시간 선택 모드

```
[날짜] 선택 (달력 or 오늘/내일/모레 칩)
    ↓
[시간대] 선택 (점심/저녁/야식 or 특정 시간)
    ↓
[인원] 선택 (1~6명+)
    ↓
캐치테이블 API → 해당 날짜 예약 가능한 식당만 필터
    ↓
도달시간순 정렬 → 바텀시트 + 지도 핀 업데이트
```

### 도달시간 + 예약시간 조합 표시

```
예약 18:30 | 지금 출발 → 18:20 도착 → "여유 있음 ✅"
예약 18:30 | 지금 출발 → 18:40 도착 → "늦을 수 있음 ⚠️"
예약 18:30 | 지금 출발 → 19:00 도착 → "도착 어려움 ❌"
```

## 05. 외부 API 연동

### 5-1. 캐치테이블 (예약 가능 여부)

| 항목 | 내용 |
|------|------|
| **도구** | `catchtable-cli` (PyPI, Python) |
| **방식** | 비공식 HTTP API (curl_cffi, Cloudflare 우회) |
| **조회 내용** | 날짜별 예약 가능 상태(AVAILABLE/CLOSED), 가능 인원수, 혜택 |
| **범위** | 오늘~14일간 (day_slots) |
| **호출 타이밍** | 사용자가 날짜 선택 시 + 캐시(TTL 30분 권장) |
| **fallback** | API 실패 시 "예약 정보 확인 불가" 표시, 전화 연결 유도 |
| **인증** | 선택적 (`CT_SESSION_COOKIE`). 미인증으로도 기본 조회 가능 |

#### 사용 예시

```bash
# 검색
ct search search "정식당" -f json
# → { "data": { "shops": [{ "code": "jungsik", ... }] } }

# 예약 가능 날짜 조회
ct shop info jungsik -f json
# → { "day_slots": { "data": [{ "date": "2026-06-27", "availableStatus": "AVAILABLE", "availablePersonCounts": [3,4] }] } }
```

### 5-2. 카카오 모빌리티 (도달시간)

## 07. 데이터 현황 (기존 유지)

### 07-1. 개요
- 총 228개 식당, 서울 강남·용산권 집중
- 소스 컬럼 7개: 식당명(한/영), 위치, 종류, 전화번호, 주소, 메모

### 07-2. 업종 정규화 (7종)

| 대표 업종 | 건수 | 이모지 |
|------|:---:|:--:|
| 한식 | 68 | 🍚 |
| 일식 | 43 | 🍣 |
| 이탈리안 | 33 | 🍝 |
| 중식 | 30 | 🥡 |
| 프렌치 | 29 | 🥐 |
| 아메리칸·양식 | 20 | 🍔 |
| 아시안·기타 | 5 | 🌏 |

### 07-3. 위치(동) 정규화
- 청담동(45), 한남동(40), 압구정동(25), 신사동(20), 논현동(11) 등
- 오타 수정 3건, 통칭(광화문·잠실 등) 그대로 유지
- 필터는 동 단위, 건수 내림차순

### 07-4. 데이터 모델

```
Restaurant      (id, name_kor, name_eng, location_norm, location_raw,
                 category_norm, category_raw, tel, address, lat, lng,
                 public_desc, hours, internal_memo, parking,
                 catchtable_alias, catchtable_matched,
                 country, updated_by, updated_at)

CatchtableCache (restaurant_id, alias, shop_ref, shop_name,
                 biz_hours, parking_guide, price_lunch, price_dinner,
                 rating, image_url, online_yn,
                 fetched_at, ttl)

DaySlotCache    (restaurant_id, date, available_status,
                 available_persons, benefit, fetched_at, ttl)

AccessConfig    (id, url_token, viewer_password_hash, owner_password_hash,
                 session_days, updated_at)

Session         (token, role, expires_at)
```

## 08. 비기능 요구사항

| 항목 | 내용 |
|------|------|
| **플랫폼** | 모바일 우선 웹뷰. 앱 배포 없음. |
| **성능** | 지도 초기 로딩 3초 이내. 예약 조회는 비동기(로딩 스피너). 도달시간은 필터 후 가시 식당만 계산. |
| **캐싱** | 캐치테이블 day_slots: TTL 30분. 도달시간: TTL 5분. 식당 상세: TTL 7일. |
| **보안** | HTTPS, 비번 해시(bcrypt), 세션 쿠키(HttpOnly/Secure), 서버 모드 판정 |
| **API 키 관리** | 카카오맵/모빌리티 API 키, 캐치테이블 세션 쿠키 → `.env.local` |
| **fallback** | 캐치테이블 API 실패 → "예약 정보 미확인" + 전화 연결. 모빌리티 실패 → 직선거리 기반 추정. |
| **디자인** | 다크 테마 + 골드 액센트. 큰 텍스트. 터치 44px 이상. |

## 09. 화면별 상세

### S1. 비밀번호 입력 (/) — 기존과 동일

- 6자리 숫자 입력 → 서버 비교 → 모드 분기 → /map 리다이렉트
- "비밀번호를 입력해 주세요" / placeholder "6자리 숫자" / 버튼 "확인"
- 실패: "비밀번호가 일치하지 않습니다."
- 잠금 없음 (데모)

### S2. 메인 — 지도 + 바텀시트 (/map)

**지도 영역 (상단 60%)**
- 카카오맵, 현위치 중심 자동 줌
- 식당 핀: 예약 가능=골드, 불가/미확인=회색, 미매칭=점선 테두리
- 핀 탭 → 간략 팝업(식당명, 도달시간, 예약 상태)

**컨트롤 바 (지도 위 오버레이)**
- [지금 바로] 버튼 (골드, 눈에 띄게)
- [날짜] 칩: 오늘 / 내일 / 모레 / 달력 아이콘(직접 선택)
- [시간대] 칩: 점심 / 저녁 / 야식
- [인원] 칩: 2명 (기본) / 1~6+

**필터 칩 (컨트롤 바 아래)**
- 업종(7종) | 지역(동) | 초기화

**바텀시트 (하단, 스와이프)**
- 접힌 상태: 상위 3~4개 식당 카드 미리보기
- 펼친 상태: 전체 필터 결과 리스트
- 각 카드: `[도달시간(분)] 식당명 · 업종 · 동 | 예약가능 ✅ / 불가 ❌ / 미확인 ？`
- 정렬: **도달시간 짧은 순** (기본)

### S3. 식당 상세 (/restaurant/[id])

**상단**
- 대표 이미지 (캐치테이블 제공, 없으면 업종 이모지)
- 식당명(한/영), 업종, 위치(동)
- 평점(캐치테이블), 가격대

**도달시간 섹션**
- "현재 위치에서 **약 N분**" (큰 폰트)
- 소형 지도에 경로 표시
- 예약시간 대비 도착 가능 여부 표시 (여유 ✅ / 위험 ⚠️)

**예약 섹션**
- 가능 날짜 목록 (14일간)
- 각 날짜: 가능 인원수 표시
- [캐치테이블에서 예약] 버튼 → 외부 이동

**기본정보**
- 영업시간, 주차, 전화, 주소
- 식당 소개 (DB + 캐치테이블 `serviceDesc`)

**액션 버튼 (하단 고정)**
- 예약 | 길찾기 | 전화 | 공유

**Owner 모드 추가**
- [편집] 버튼
- 🔒 자유 메모 표시

### S4. 식당 편집 (/restaurant/[id]/edit) — Owner 전용

- 메모(🔒), 소개·영업시간·주차(📢)
- 카테고리 드롭다운
- 캐치테이블 매칭: alias 수동 지정/변경/해제
- 🔒/📢 배지

### S5. 관리 (/settings) — Owner 전용

- 비밀번호 변경(임원용/사장님)
- URL/QR 재발급
- 전체 세션 강제 만료

## 11. 기술 구현 개요

### API 호출 최적화

```
[사용자 날짜 선택]
    ↓
[서버] 228개 중 catchtable_matched=true인 식당 (208건)
    ↓  캐시 확인: DaySlotCache에 해당 날짜 데이터 있고 TTL 이내?
    ├ 있음 → 캐시 반환
    └ 없음 → catchtable-cli 호출 → 캐시 저장 → 반환
    ↓
[클라이언트] 예약 가능 식당 필터링
    ↓
[서버] 필터 후 가시 식당(~20~50건)만 카카오 모빌리티 호출
    ↓  캐시 확인: 5분 이내 동일 출발점-도착점?
    ├ 있음 → 캐시 반환
    └ 없음 → 모빌리티 API 호출 → 캐시 저장 → 반환
    ↓
[클라이언트] 시간순 정렬 → 지도 핀 + 바텀시트 갱신
```

### 캐치테이블 배치 조회 전략

- 228건 전체를 매시간 배치 조회하는 것은 비효율
- **Lazy 조회**: 사용자가 날짜를 선택할 때 해당 날짜의 가용 상태를 조회
- **Pre-warm**: 오늘/내일 데이터는 서버 기동 시 + 30분마다 갱신
- **개별 조회**: 식당 상세 진입 시 해당 식당 14일간 데이터 갱신
